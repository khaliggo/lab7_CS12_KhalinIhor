# Лабораторна робота №7 — Відображення у пам'ять (mmap IPC)

---

## Про проєкт

Цей проєкт виконаний в рамках **Лабораторної роботи №7** з теми _«Відображення у пам'ять»_.

Мета роботи — дослідити механізми міжпроцесної взаємодії (IPC) в Linux через:
- **`mmap`** — відображення файлу або анонімної ділянки пам'яті в адресний простір процесу
- **`fcntl.flock()`** та **`fcntl.lockf()`** — файлові блокування для організації взаємовиключного та розділеного доступу
- **`pickle`** — бінарна серіалізація Python-об'єктів для передачі структурованих даних через спільну пам'ять

Проєкт містить два незалежних рішення (`task1.py` та `task2.py`), кожне з яких демонструє різний підхід до організації спільної пам'яті та синхронізації між процесами.

**Вимоги до середовища:** Python 3.8+, Linux (використовує `os.fork()`, `fcntl`, POSIX `mmap`)

## Як працює код

### `task1.py` — Анонімний `mmap` + `fcntl.flock()`

#### Концепція

Батьківський процес створює **анонімне відображення пам'яті** (`mmap(-1, size)`) — ділянку RAM без прив'язки до файлу. Після `os.fork()` ця ділянка успадковується всіма дочірніми процесами і залишається **спільною** між ними. Для координації запису використовується **окремий порожній файл** на диску, виключно як об'єкт для `flock`-блокування.

#### Аргументи командного рядку

| Аргумент | За замовч. | Опис |
|----------|-----------|------|
| `-n`, `--num-children` | `4` | Кількість дочірніх процесів |
| `-s`, `--mmap-size` | `65536` | Розмір mmap-регіону в байтах |
| `--min-str` | `5` | Мінімальна довжина рядка в повідомленні |
| `--max-str` | `30` | Максимальна довжина рядка в повідомленні |

#### Структура пам'яті

```
Байти 0..3     — uint32 big-endian: кількість зайнятих байт після заголовку
Байти 4..N     — Record 1: [4 байти довжини][pickle payload]
Байти N+1..M   — Record 2: [4 байти довжини][pickle payload]
...
```

Перші 4 байти (`HEADER_SIZE = 4`) — це лічильник, який показує скільки байт після заголовку вже зайнято. Кожен дочірній процес читає цей лічильник, щоб знайти кінець зайнятої зони, дописує свій запис туди, і оновлює лічильник — все це атомарно під блокуванням.

#### Покроковий алгоритм

**Батьківський процес (`_main`):**
1. Створює тимчасовий порожній файл (`tempfile.mkstemp`) — він використовується тільки як маркер для `flock`, дані в нього не пишуться
2. Створює анонімний `mmap(-1, mmap_size)` та записує `0` у перші 4 байти (заголовок — початковий стан: 0 байт зайнято)
3. Викликає `os.fork()` N разів, зберігаючи PID кожного дочірнього процесу
4. Очікує завершення всіх дочірніх процесів через `os.waitpid()`
5. Читає всі повідомлення з mmap викликом `read_all_messages()` та виводить їх

**Дочірній процес (`child_worker`):**
1. Генерує повідомлення-словник: `{'pid': <pid>, 'str': <random_string>}`
2. Серіалізує його через `pickle.dumps()`
3. Відкриває lock-файл батька (`os.open`)
4. Викликає `write_message()` і після завершення закриває fd та завершується через `os._exit(0)`

**Запис у спільну пам'ять (`write_message`):**
1. `fcntl.flock(lock_fd, LOCK_EX)` — захоплює ексклюзивне блокування на lock-файл. Інші процеси блокуються тут і чекають
2. Читає поточне значення лічильника з байт `[0:4]` mmap
3. Перевіряє, чи вистачає місця: `HEADER_SIZE + used + len(record) <= mmap_size`
4. Записує `record = struct.pack('>I', len(payload)) + payload` в позицію `HEADER_SIZE + used`
5. Оновлює лічильник: записує `used + len(record)` назад у байти `[0:4]`
6. `fcntl.flock(lock_fd, LOCK_UN)` — знімає блокування, наступний процес у черзі може заходити

**Читання всіх повідомлень (`read_all_messages`):**
1. Читає лічильник `used` з заголовку
2. В циклі проходить по зайнятій зоні: зчитує 4-байтовий заголовок довжини, потім `payload_len` байт payload
3. Кожен payload десеріалізує через `pickle.loads()` і додає до списку
4. Зупиняється коли `pos >= used`

#### Приклад запуску

```bash
python3 task1.py                                   # За замовчуванням: 4 дочірніх, 64KB mmap
python3 task1.py -n 6                              # 6 дочірніх процесів
python3 task1.py -n 8 -s 131072                   # 8 процесів, 128KB mmap
python3 task1.py -n 3 --min-str 10 --max-str 50   # Довші рядки в повідомленнях
python3 task1.py --help                            # Показати всі аргументи
```

### `task2.py` — Файловий `mmap` + `fcntl.lockf()`

#### Концепція

Батьківський процес створює **файл фіксованого розміру** на диску та відображає його в пам'ять. Файл розділений на **N рівних слотів** — кожен дочірній процес отримує свій слот за індексом і пише тільки туди. Синхронізація здійснюється через `fcntl.lockf()` з явним зазначенням байтового діапазону (`offset` + `length`), що дозволяє різним процесам паралельно блокувати **різні** ділянки одного файлу. Батьківський процес **циклічно поллить** усі слоти зі спільним (`LOCK_SH`) блокуванням.

#### Константи

| Константа | Значення | Опис |
|-----------|---------|------|
| `NUM_CHILDREN` | `3` | Кількість дочірніх процесів / слотів |
| `SLOT_SIZE` | `256` | Розмір одного слоту в байтах |
| `FILE_SIZE` | `768` | Загальний розмір файлу (`NUM_CHILDREN * SLOT_SIZE`) |
| `FILE_NAME` | `mmap_slots.bin` | Ім'я файлу-відображення |
| `READ_INTERVAL` | `0.2` | Інтервал поллінгу батьком (секунди) |

#### Структура файлу `mmap_slots.bin`

```
Байти   0..255  — Слот 0: [4Б довжина][pickle payload][нулі до кінця слоту]
Байти 256..511  — Слот 1: [4Б довжина][pickle payload][нулі до кінця слоту]
Байти 512..767  — Слот 2: [4Б довжина][pickle payload][нулі до кінця слоту]
```

Всередині кожного слоту перші 4 байти — big-endian uint32 з реальною довжиною pickle-payload. Значення `0` означає, що слот ще порожній (початковий стан файлу — суцільні нулі).

#### Покроковий алгоритм

**Батьківський процес (`_main`):**
1. Створює файл `mmap_slots.bin` заповнений нулями через `create_mmap_file()`: відкриває з флагами `O_RDWR | O_CREAT | O_TRUNC`, записує `FILE_SIZE` нульових байт
2. Відображає файл у пам'ять: `mmap.mmap(fd, 0)` (розмір `0` = весь файл)
3. Форкає N дочірніх процесів, кожному передаючи його `slot_index`
4. Запускає `parent_poll()` — циклічне читання всіх слотів поки не зберуться всі повідомлення
5. Чекає завершення дочірніх через `waitpid()`
6. Виводить зібрані повідомлення, закриває mmap та видаляє файл

**Дочірній процес (`child_write`):**
1. Відкриває `mmap_slots.bin` власним `fd` (незалежно від батька)
2. Створює повідомлення: `{'pid': pid, 'slot': slot_index, 'text': '...'}`
3. Серіалізує через `pickle.dumps()`, перевіряє що `4 + len(payload) <= SLOT_SIZE`
4. Захоплює ексклюзивне блокування **тільки свого слоту**: `fcntl.lockf(fd, LOCK_EX, SLOT_SIZE, start, SEEK_SET)` — де `start = slot_index * SLOT_SIZE`
5. Формує запис: `length_bytes (4Б) + payload + нульове заповнення до SLOT_SIZE`
6. Пише через `mem.seek(start)` → `mem.write(padded)` → `mem.flush()`
7. Знімає блокування: `fcntl.lockf(fd, LOCK_UN, SLOT_SIZE, start, SEEK_SET)`
8. Закриває `mem` та `fd`, завершується через `os._exit(0)`

**Батьківський поллінг (`parent_poll`):**
1. В циклі перебирає всі слоти, які ще не були прочитані
2. Для кожного слоту викликає `read_slot()` зі спільним блокуванням (`LOCK_SH`)
3. Якщо `payload_len > 0` — слот містить повідомлення, десеріалізує і зберігає
4. Між раундами засинає на `READ_INTERVAL` секунд
5. Завершується коли зібрані всі N повідомлень або вичерпано `timeout`

**Читання слоту (`read_slot`):**
1. `fcntl.lockf(fd, LOCK_SH, SLOT_SIZE, start, SEEK_SET)` — спільне блокування: читати можуть одночасно кілька процесів, але не одночасно з `LOCK_EX`
2. `mem.seek(start)` → `mem.read(SLOT_SIZE)` — читає весь слот
3. `fcntl.lockf(fd, LOCK_UN, ...)` — знімає блокування
4. Перші 4 байти → `payload_len`; якщо `0` — повертає `None`
5. Інакше `pickle.loads(raw[4 : 4 + payload_len])` → Python-об'єкт

#### Ключова відмінність `lockf` vs `flock`

`fcntl.lockf()` підтримує **byte-range locking** — блокування конкретного байтового діапазону файлу. Це означає що дочірній процес 0 може одночасно тримати `LOCK_EX` на байти `0..255`, поки дочірній процес 1 тримає `LOCK_EX` на байти `256..511` — вони не заважають одне одному, бо їхні діапазони не перетинаються.

#### Приклад запуску

```bash
python3 task2.py    # Запустити з параметрами за замовчуванням (3 слоти, 256Б кожен)
```

## Порівняння двох підходів

| Характеристика | `task1.py` | `task2.py` |
|----------------|-----------|-----------|
| Тип mmap | Анонімний (`mmap(-1, ...)`) | Файловий (`mmap(fd, 0)`) |
| Механізм блокування | `fcntl.flock()` — весь файл | `fcntl.lockf()` — діапазон байт |
| Структура пам'яті | Динамічна черга (append) | Фіксовані слоти |
| Granularity блокування | Один lock на всіх | Окремий lock на кожен слот |
| Паралелізм запису | Послідовний (один за одним) | Паралельний (різні слоти одночасно) |
| Читання батьком | Один прохід після `waitpid` | Поллінг під час роботи дітей |
| Persistence | Зникає разом з процесом | Файл існує на диску під час роботи |
